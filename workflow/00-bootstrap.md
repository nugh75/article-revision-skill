# 00 — Bootstrap

Run **first** when the skill is invoked. Idempotent: if all artifacts are already in place, exits in seconds and proceeds to `10-setup.md`.

The bootstrap establishes the **revision environment**: Python venv with the required packages, the `.bib` file, the `.env` with credentials and editorial parameters, and the editorial norms file. Each step asks the user before creating anything — never write files silently.

## Step A — Python virtual environment

1. Look for a venv at `<project-root>/.venv/`.
2. If absent, ask the user:

   > Non trovo un venv per gli script della skill. Vuoi che ne crei uno in `.venv/` e installi i pacchetti da `requirements.txt`?
   > (sì / no / specifica path alternativo)

3. On confirmation:

   ```bash
   python3 -m venv .venv
   .venv/bin/python -m pip install --upgrade pip
   .venv/bin/python -m pip install -r .claude/skills/article-revision/requirements.txt
   ```

4. If the user refuses venv creation, ask before falling back:

   > Vuoi usare Python di sistema solo per questa sessione? Alcuni script potrebbero non funzionare se mancano pacchetti.
   > (sì / no)

   If the answer is `no`, skip Python-based checks and note that scripts are unavailable until a venv or interpreter is configured.

5. If a venv already exists, verify the packages match `requirements.txt`:

   ```bash
   .venv/bin/python -m pip check
   .venv/bin/python -m pip install -r .claude/skills/article-revision/requirements.txt --upgrade-strategy only-if-needed
   ```

6. Remember the interpreter for the session:

   - venv accepted/found: `PYTHON_BIN=.venv/bin/python`
   - alternative path accepted: `PYTHON_BIN=<custom-path>/bin/python` or the explicit interpreter path
   - system Python accepted: `PYTHON_BIN=$(command -v python3)`

7. Ask before persisting `PYTHON_BIN` to `.env`:

   > Vuoi salvare `PYTHON_BIN=<path>` in `.env` per le prossime sessioni?
   > (sì / no)

8. All subsequent Python scripts in this workflow must be run with `PYTHON_BIN`; never call `python3 scripts/...` directly unless the user explicitly chose system Python.

## Step B — `.bib` file

1. Read `BIBLIOGRAPHY_BIB_PATH` from `.env`. If `.env` doesn't exist yet, default to `bibliografia/reference.bib`.
2. If the path doesn't exist, ask:

   > Non trovo `<path>/reference.bib`. Vuoi che ne crei uno vuoto qui? (sì / specifica path alternativo)

3. On confirmation, create the directory if needed and write a minimal `.bib`:

   ```bibtex
   % Bibliography file generated by article-revision skill on YYYY-MM-DD
   % Add entries below using the bibliography skill (e.g. praxis-bibliography-citations).
   ```

## Step C — `.env` file

1. Look for a `.env` at the project root.
2. If absent or missing required keys, walk through the keys interactively. For each missing/empty key, ask one question (or batch — see below). Available keys, in priority order:

   **Required**

   - `EDITORIAL_LIMIT_CHARS` (or `EDITORIAL_LIMIT_WORDS`).
     Ask: *"Qual è il limite editoriale della rivista? Inserisci numero di caratteri (default `40000`) oppure 'words' seguito dal numero (es. `words 8000`)."*
   - `EDITORIAL_NORMS_PATH`.
     Ask: *"Dove si trova il file delle norme redazionali? (lascia vuoto per usare un placeholder, vedi step D)"*
   - `BIBLIOGRAPHY_BIB_PATH`.
     Default: il path scelto in step B.

   **Optional but recommended**

   - `CROSSREF_USER_AGENT` and `OPENALEX_USER_AGENT` (form: `ProjectName/1.0 (mailto:author@example.org)`).
   - `ZOTERO_USER_ID`, `ZOTERO_API_KEY`, `ZOTERO_GROUP_ID` (per la sincronizzazione bibliografica). Ask explicitly:
     > Vuoi configurare la sincronizzazione Zotero? Servono `user_id`, `api_key` e `group_id`. Posso saltare e lo configurerai in seguito.
   - `ARTICLE_LANG` — di solito vuoto (auto-detection).
   - `PYTHON_BIN` — interprete Python per gli script della skill; di solito `.venv/bin/python`.
   - `AUTO_BUMP_THRESHOLD` — quante modifiche accettate consecutive prima che la skill proponga un bump di versione (default `5`).

3. Append the new keys to `.env` (or create the file). Never overwrite existing values without asking.

4. Re-load `.env` after the edits and continue.

### Asking efficiently

If `.env` is missing several keys, batch the questions into one chat block:

```
Per completare l'ambiente mi servono questi dati. Rispondi numerando, oppure lascia vuoto per saltare:
1) Limite caratteri (default 40000):
2) Path file norme redazionali:
3) Path file bibliografia (default bibliografia/reference.bib):
4) Crossref/OpenAlex User-Agent (formato `Project/1.0 (mailto:you@host)`):
5) Zotero — user_id, api_key, group_id (separati da virgola):
6) Python interpreter (default `.venv/bin/python`):
7) Soglia auto-bump versione (default 5):
```

## Step D — Editorial norms file

1. After `EDITORIAL_NORMS_PATH` is set, check whether the file exists.
2. If absent, ask:

   > Il file delle norme redazionali (`<path>`) non esiste. Vuoi:
   > - **(a)** crearne uno vuoto (placeholder, lo riempi a mano)
   > - **(b)** copiarlo da un altro progetto (specifica il path sorgente)
   > - **(c)** rimandare e procedere senza norme (la skill funzionerà ma non potrà far valere le regole della rivista)

3. Apply the choice. For (a):

   ```markdown
   # Norme redazionali — <rivista>

   <!-- Inserire qui le regole binding della rivista:
        - lunghezza massima
        - stile citazioni
        - virgolette, italici, neretti
        - struttura sezioni
        - formato bibliografia
   -->
   ```

## Step E — Project structure

Verify or create:

```
<project-root>/
├── articoli/                       # if absent, create
└── revisioni/                      # if absent, create
```

Don't move existing files into these folders without asking.

After all bootstrap steps, output one chat block:

```
✅ Bootstrap completo

- venv: <.venv/ | system>
- python: <PYTHON_BIN | unavailable>
- bibliografia: <path> (X voci)

Procedo con `10-setup.md` per individuare l'articolo attivo? (sì / no)
```

If everything was already in place, this block can be condensed to one line: *"✅ Ambiente già configurato. Procedo con il setup."*.
